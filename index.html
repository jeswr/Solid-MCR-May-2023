A Solid foundation: What do we have today?

  • LDP "a file system for the Web"
    • Uses HTTP(S)
    • Performs CRUD operations on resources (documents)
    • Resources can be RDF & non-RDF
    • Resources are organised in containers (folders)
  • Authorisation (ACP / WAC)
    • Permission agents & applications to read, write, append and delete resources

Where are some difficulties?

  • Interoperability
    • Applications cannot understand data written by other apps
    • Applications cannot find data written by other apps
  • Performance
    • Apps struggle to work with "big data"
  • Requesting & automating authorisation

What solutions can we use to find data today?

  • client-to-client agreements
    • Type Indexes
    • Closed ecosystem assumptions
  • brute-force discovery
    • Pod-wide link-traversal
      • Use tools like Comunica
      • Optimize with services like the ESS Fragments Indexer (but don't rely on them)

What solutions can we use to understand data today?

  • client-to-client standards
    • agree on shapes
  • advanced mode
    • client-side reasoning with:
      • eye
      • N3.js
      • Comunica

Which difficulties can we solve today?

  • Interoperability
    • For finding data we have type-indexes, but they can be corrupted as they are client-maintained
    • For understanding data we can agree to use the same shapes; but we do not access the power of RDF semantics
  • Performance
  • Requesting & automating authorisation

Whats on the horizon?

  • Interoperability
    • Data views such as
      • a server managed type index
      • virtualising data by shape
  • Performance
    • Data views for query optimization
  • Requesting & automating authorisation
    • Auxilliary resource views on "interesting" policies

What are these views of?

  • Short term
    • The LDP storage

What are these views of?

  • Long term

      • Application Data
      • Provenance (metadata)
      • Policies

Envisioning the far future: A Web of Rules and Data

  • No application logic
  • No code
  • Just a (socratic) reasoning dialogue

What does this dialogue look like?

  • Set a goal
    • I want my fridge to be well stocked, with healthy food that I like, whenever I am home.
  • Communicate (using RDF over HTTP) your goal, or sub-goals to relevant stakeholders.
    • By sending questions formulated in RDF from your reasoning engine to reasoners hosted by the Supermarket and Delivery Service.

What does this dialogue look like?

  • Stakeholders respond with partial answers, or questions, permitted by their policies.
    • Here are the prices of the items you are interested in. Please tell me the time you want to pick it up so I can tell you the stock I will have at that time.
  • Rince & repeat

Envisioning the far future: A Web of Rules and Data

      [...] Similarly, reasoners will start rewiring software for specific situations. EYE already does this when performing rule-based Web service composition."
